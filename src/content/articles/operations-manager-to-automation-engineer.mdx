Nobody plans to become an automation engineer. Most people who are good at it came from somewhere else — operations, finance, marketing, admin — and started automating out of frustration.

That's my story.

## The operations side

For several years I worked as an operations manager. My job was making things run smoothly: coordinating teams, managing workflows, hitting targets, putting out fires.

I was good at it. But the actual work was exhausting in a specific way — a lot of it was moving information from one place to another. Updating spreadsheets. Sending the same emails. Compiling the same reports. Copying data between systems that should have talked to each other automatically.

I kept thinking: this shouldn't require a human.

## The first automation

The first workflow I built was embarrassingly simple: a Google Forms submission triggered a Zapier action that added a row to a Google Sheet and sent me an email.

Two steps. No code. Twenty minutes to build.

It replaced a manual process I'd been doing every day for eight months.

The feeling when I tested it and watched the row appear in the sheet — without me doing anything — was something I haven't forgotten. Not because it was technically impressive. Because it proved the principle: repetitive work is automatable.

## The learning curve

From Zapier I moved to Make (formerly Integromat) and eventually n8n. Each tool added capability. n8n was the inflection point — self-hosted, no execution limits, actual programming-adjacent logic.

The skills I had to learn from scratch:
- Basic SQL (MySQL — enough to write SELECT, INSERT, UPDATE queries)
- Reading API documentation
- JSON structure and how data flows between systems
- Webhook patterns (how to receive and send HTTP requests)
- Basic JavaScript (mostly for Code nodes in n8n)

The skills that transferred directly from operations:
- Process documentation (understanding a workflow before automating it)
- Thinking in edge cases (what happens when this breaks?)
- Stakeholder communication (explaining technical things to non-technical people)
- Prioritisation (which automations have the highest ROI?)

Operations management teaches you to see systems. That turns out to be most of what automation engineering is.

## What actually changed day-to-day

Before: I spent 40% of my working hours on tasks that required my hands but not my brain — data entry, report compilation, status updates, manual follow-ups.

After: those tasks run automatically. I spend that 40% on actual problems — things that require judgment, context, and human understanding.

I didn't work less. I worked on better things.

## The income side

I'm not going to give specific numbers, but the transition made a meaningful difference financially. Automation skills are genuinely scarce in Malaysia and Southeast Asia specifically. Most companies know they should automate but don't have the in-house capability to do it.

Being able to build systems that reduce manual work has a clear, calculable ROI. That makes it easier to price consulting work and justify investment.

## What I'd tell someone in the position I was in

If you're an operations person who thinks "I could automate this" — you're probably right. And you're further along than you think.

The gap between "understanding how a process works" and "automating that process" is mostly just tool familiarity. You already have the harder skill: knowing what needs to happen and in what order.

Start with the most painful repeating task you have. Build a simple automation for it. It doesn't need to be perfect. It just needs to work.

From there, the path builds itself.
